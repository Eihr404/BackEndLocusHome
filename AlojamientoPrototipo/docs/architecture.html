1. Arquitectura General del Sistema

El backend de AlojamientoMR está estructurado como un ecosistema descentralizado de microservicios. Cuenta con una puerta de enlace central (Gateway) y comunicación tanto sincrónica (gRPC) como asincrónica (RabbitMQ/CloudAMQP).


               [ CLIENTE / FRONTEND ]
                         │
                         ▼ (HTTPS REST / JSON)
                  [ ApiGateway ] (YARP)
            ┌────────────┼────────────┬────────────┐
            ▼            ▼            ▼            ▼
      [ Alojamientos ] [ Reservas ] [ Usuarios ] [ Facturacion ]
            ▲            │
            └────────────┘
              (gRPC-Web) 
            
      [------ Event Bus (CloudAMQP / RabbitMQ) ------]
          - Facturacion publica: FacturaPagadaEvent
          - Reservas consume el evento para actualizar estado

2. Detalle de los Microservicios

Cada microservicio opera con su propio contexto de negocio, base de datos aislada y responsabilidades definidas:

🏠 Alojamientos API

Gestiona los datos de los hoteles, cabañas o alojamientos disponibles y su calendario de ocupación.

  • Puerto Local: 5002
  • Ruta API: /api/v1/alojamientos
  • Servicio gRPC: CalendarioGrpcService para validar fechas libres.

📅 Reservas API

Orquesta la creación de reservas de clientes. Hace una llamada sincrónica vía gRPC a Alojamientos para validar la disponibilidad antes de confirmar.

  • Puerto Local: 5003
  • Ruta API: /api/v1/reservas
  • Consumidor de Eventos: Recibe notificaciones de facturas cobradas para actualizar el estado del alquiler.

👤 Usuarios API

Administra perfiles de usuarios, roles y datos personales.

  • Puerto Local: 5004
  • Ruta API: /api/v1/usuarios

💳 Facturación API

Se encarga de generar facturas cuando se confirma una reserva y publicar eventos de pago completado.

  • Puerto Local: 5005
  • Ruta API: /api/v1/facturacion

3. Configuración de Base de Datos (PostgreSQL)

Para cumplir con los principios de microservicios, cada servicio tiene su propia base de datos dedicada. En producción se utiliza PostgreSQL (por ejemplo, alojado en Supabase, Neon o Render PostgreSQL).

¡Atención Compañeros de Equipo!

NO compartan las mismas tablas ni la misma base de datos. Cada uno debe crear un recurso de base de datos independiente (en su cuenta de Supabase u otro proveedor) y utilizar el nombre de la cadena de conexión adecuado.

Claves de Connection String en appsettings.json:

Deben configurar la cadena de conexión correspondiente para cada servicio bajo la sección ConnectionStrings:

Proyecto / Microservicio Clave de Configuración Proveedor Recomendado
Alojamientos.API ConexionAlojamientos PostgreSQL (Supabase) con soporte PostGIS/Extensiones si aplica
Reservas.API ConexionReservas PostgreSQL (Supabase)
Facturacion.API ConexionFacturacion PostgreSQL (Supabase)
Usuarios.API ConexionUsuarios PostgreSQL (Supabase + NetTopologySuite para geolocalización)

Plantilla de Cadena de Conexión (Sin Credenciales Reales):

"ConnectionStrings": {
  "Conexion[NombreServicio]": "Host=[TU_HOST_AQUÍ];Port=[TU_PUERTO_AQUÍ];Database=[TU_BD_AQUÍ];Username=[TU_USUARIO_AQUÍ];Password=[TU_CONTRASEÑA_AQUÍ];SSL Mode=Require;Trust Server Certificate=true;Include Error Detail=true"
}

4. Integración con CloudAMQP (RabbitMQ)

La mensajería asincrónica está implementada utilizando MassTransit con el broker RabbitMQ. En la nube utilizamos CloudAMQP (plan gratuito Lemur).

¿Cómo funciona la integración?

  1. Cuando se realiza un pago o cambio en Facturacion.API, se emite un evento FacturaPagadaEvent.
  2. Reservas.API cuenta con un FacturaPagadaConsumer que escucha la cola y actualiza el estado de la reserva automáticamente sin acoplamiento directo entre APIs.

Configuración en el código:

Tanto en Reservas.API como en Facturacion.API, la conexión busca la clave de cadena de conexión llamada RabbitMQ:

"ConnectionStrings": {
  "RabbitMQ": "amqps://[USER]:[PASSWORD]@[HOST]/[VIRTUAL_HOST]"
}
Pasos para crear tu propio CloudAMQP:
  1. Regístrate gratis en CloudAMQP.
  2. Crea una nueva instancia en la región más cercana a tus servidores de Render (ej. US-East-1).
  3. Copia la AMQP URL (comienza con amqps://).
  4. Pégala en tu variable de entorno o archivo de configuración local.

5. Despliegue en Render

Para subir el ecosistema a Render, cada desarrollador del equipo debe crear sus propios servicios en su cuenta de Render. Sigue los siguientes lineamientos para evitar colisiones y fallas de comunicación:

Patrón de Nombres Recomendado para tus APIs:

Dado que los nombres en Render deben ser únicos globalmente en el subdominio *.onrender.com, personaliza los nombres agregando tus iniciales o nombre como prefijo:

  • [tu-nombre]-alojamientos-api
  • [tu-nombre]-reservas-api
  • [tu-nombre]-usuarios-api
  • [tu-nombre]-facturacion-api
  • [tu-nombre]-apigateway

Configuración del Dockerfile en Render:

El repositorio ya incluye Dockerfiles optimizados para cada microservicio. Al crear el Web Service en Render:

  1. Define el **Runtime** como Docker.
  2. Especifica la ruta del Dockerfile relativo a la raíz del repositorio, por ejemplo:
    • Para Alojamientos: Dockerfile.Alojamientos
    • Para Reservas: Dockerfile.Reservas
    • Para Gateway: Dockerfile.ApiGateway
  3. Usa la rama main para el auto-despliegue.
Configuración Crucial de gRPC-Web en Entornos Gratuitos de Render

Los balanceadores de carga gratuitos de Render **no admiten peticiones HTTP/2 gRPC estándar** directamente sobre Internet (las fuerzan a HTTP/1.1 y causan error Content-Type 'application/grpc' is not supported).

Para solucionarlo de forma permanente, configuramos gRPC-Web. Por lo tanto:

  • Alojamientos (Servidor): Expone su gRPC-Web de manera pública en la misma URL que sus endpoints REST.
  • Reservas (Cliente): Apunta a la URL de Alojamientos usando la URL pública HTTPS:
    GrpcUrls__Alojamientos = https://[tu-nombre]-alojamientos-api.onrender.com

6. Configuración de YARP Gateway

El Gateway actúa como el único punto de entrada público para los clientes. Utiliza YARP (Yet Another Reverse Proxy) y redirige el tráfico según el patrón de ruta solicitado.

Configuración en appsettings.json del Gateway:

El Gateway traduce las rutas del estilo /api/v1/mathias-rivera/... hacia el microservicio interno correspondiente. Si cambias el nombre de tus servicios en Render, debes modificar las direcciones de los clusters en el appsettings.json del Gateway:

"ReverseProxy": {
  "Clusters": {
    "reservas-cluster": {
      "Destinations": {
        "destination1": {
          "Address": "https://[tu-nombre]-reservas-api.onrender.com/"
        }
      }
    },
    "alojamientos-cluster": {
      "Destinations": {
        "destination1": {
          "Address": "https://[tu-nombre]-alojamientos-api.onrender.com/"
        }
      }
    }
  }
}

O configurarlos mediante variables de entorno en el panel de Render del Gateway:

  • ReverseProxy__Clusters__reservas-cluster__Destinations__destination1__Address = https://[tu-nombre]-reservas-api.onrender.com/
  • ReverseProxy__Clusters__alojamientos-cluster__Destinations__destination1__Address = https://[tu-nombre]-alojamientos-api.onrender.com/

8. Contexto Rápido para Modelos de Lenguaje / AIs

Si tus compañeros de equipo necesitan ayuda de un asistente de IA (como ChatGPT, Claude o Gemini) para extender el backend, pueden copiar y enviarle el siguiente texto como contexto:

"Hola Asistente de IA. Estoy trabajando en un proyecto de microservicios de .NET 8 llamado AlojamientoMR. El sistema consta de:
1. Alojamientos API: Puerto 5002, gestiona hoteles y disponibilidad mediante gRPC-Web (servicio CalendarioGrpcService) y REST. Usa la base de datos 'ConexionAlojamientos' en PostgreSQL.
2. Reservas API: Puerto 5003, consume CalendarioGrpcService a través de un cliente gRPC-Web direccionado por la variable de entorno 'GrpcUrls:Alojamientos' que apunta a la URL pública del microservicio de Alojamientos. Usa la base de datos 'ConexionReservas'.
3. Usuarios API: Puerto 5004, base de datos 'ConexionUsuarios'.
4. Facturacion API: Puerto 5005, base de datos 'ConexionFacturacion'.
5. ApiGateway: YARP Proxy que mapea rutas públicas de tipo `/api/v1/mathias-rivera/[nombre-servicio]` hacia las URL correspondientes de cada microservicio en la nube.
6. Event Bus: MassTransit con RabbitMQ mediante la conexión 'ConnectionStrings:RabbitMQ' provista por CloudAMQP. Facturacion publica FacturaPagadaEvent que es escuchado de forma asíncrona por el FacturaPagadaConsumer en Reservas API.
Toda la configuración de variables de entorno usa doble guion bajo (__) para simular la jerarquía de appsettings.json en producción de Render (ej: ConnectionStrings__ConexionReservas, GrpcUrls__Alojamientos). Ayúdame a agregar nuevas características manteniendo esta arquitectura."
