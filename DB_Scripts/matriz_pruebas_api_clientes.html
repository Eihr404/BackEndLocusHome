HU-01 Registrar cliente
Como operador, quiero registrar un nuevo cliente con sus datos básicos para poder gestionarlo en el sistema.
  • El sistema asigna un ID único automático
  • Se validan nombre, email y teléfono obligatorios
  • No se permiten emails duplicados
  • Se retorna 201 con el recurso creado
  • El timestamp de creación queda registrado
 POST /api/v1/clientes TC-F01, TC-F02, TC-F03
HU-02 Consultar clientes
Como operador, quiero consultar uno o varios clientes para visualizar su información actualizada.
  • Se puede obtener un cliente por su ID
  • Se retorna 404 si el ID no existe
  • El listado soporta paginación (page/size)
  • Se pueden filtrar por nombre o estado
  • La respuesta incluye metadatos de paginación
 GET /api/v1/clientes
GET /api/v1/clientes/{id}		 TC-F04, TC-F05, TC-F06, TC-F07
HU-03 Actualizar datos del cliente
Como operador, quiero modificar los datos de un cliente existente cuando la información cambia.
  • PUT actualiza todos los campos del cliente
  • PATCH actualiza solo los campos enviados
  • Se retorna el recurso actualizado
  • No se permite cambiar el ID del cliente
  • Se registra timestamp de última modificación
 PUT /api/v1/clientes/{id}
PATCH /api/v1/clientes/{id}		 TC-F08, TC-F09
HU-04 Eliminar cliente
Como administrador, quiero eliminar un cliente del sistema cuando ya no sea necesario.
  • Solo usuarios con rol admin pueden eliminar
  • Se retorna 204 tras eliminación exitosa
  • Se retorna 404 si el cliente no existe
  • El cliente no puede recuperarse tras la eliminación
 DELETE /api/v1/clientes/{id} TC-F10, TC-F11
HU-05 Gestionar estado del cliente
Como operador, quiero activar o desactivar un cliente sin eliminarlo para controlar su acceso al sistema.
  • Los estados válidos son: activo, inactivo, suspendido
  • El cambio de estado queda auditado con usuario y timestamp
  • Un cliente inactivo no puede autenticarse en otros servicios
  • Se retorna el estado actualizado en la respuesta
 PATCH /api/v1/clientes/{id}/estado TC-F12
HU-06 Acceso seguro a la API
Como desarrollador, quiero que todos los endpoints estén protegidos con JWT para garantizar que solo usuarios autenticados y autorizados operen sobre los datos.
  • Toda petición requiere JWT válido en header Authorization
  • Los tokens expiran según TTL configurado
  • El rol del usuario determina las operaciones permitidas
  • Toda comunicación ocurre por HTTPS
  • Los orígenes permitidos están definidos en CORS
  • Toda entrada es validada y sanitizada
