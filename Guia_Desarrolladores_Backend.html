Guía del Backend: AlojamientoMR

¡Hola equipo! Este documento explica la arquitectura del backend, cómo inicializar la base de datos y, lo más importante, dónde y cómo cambiar las cadenas de conexión (URLs) de las bases de datos para trabajar localmente o desplegar en producción.

1. Arquitectura General

El backend está construido usando .NET 8 (C#) con una arquitectura de Monolito Modular (Clean Architecture). Esto significa que el código está dividido en 4 módulos independientes, cada uno conectado a su propia base de datos PostgreSQL (idealmente en Supabase):

  • Usuarios: Gestiona el registro de clientes y perfiles.
  • Alojamientos: Catálogo de hoteles, suites y disponibilidad.
  • Reservas: Motor transaccional de check-ins y check-outs.
  • Facturación: Pagos y comprobantes.

2. Scripts de Base de Datos

En la carpeta DB_Scripts se encuentran los archivos SQL necesarios para crear las tablas, funciones y triggers. Debido a las relaciones lógicas, debes ejecutarlos estrictamente en este orden en tu gestor de base de datos (PgAdmin, DBeaver o el panel de Supabase):

1
01_DB_Usuarios.sql: Crea roles, clientes y procedimientos de registro.
2
02_DB_Alojamientos.sql: Crea tipos, habitaciones y el calendario crítico de disponibilidad.
3
03_DB_Reservas.sql: Crea lógica de reservas y orquestación de fechas.
4
04_DB_Facturacion.sql: Crea métodos de pago y auditoría general.

3. ⚠️ ¿Dónde configurar las URLs de las Bases de Datos?

Para que la API de .NET se pueda conectar a Supabase (o a tu PostgreSQL local), necesitas configurar 4 Connection Strings (Cadenas de conexión). Hay dos formas de hacerlo: localmente para desarrollo o mediante Variables de Entorno en producción (Render/Azure).

Modo Local (Para pruebas en tu PC)

Abre el archivo appsettings.json que está dentro de la carpeta del proyecto Microservicio.Clientes.Api.

Busca la sección ConnectionStrings. Debería verse exactamente así (solo reemplaza la contraseña y el host si cambias de base):

"ConnectionStrings": {
  "ConexionUsuarios": "Host=aws-0-us-east-1.pooler.supabase.com;Port=6543;Database=postgres;Username=postgres.fcdwquzctjdxqssgczlq;Password=TU_CONTRASEÑA_AQUI",
  "ConexionAlojamientos": "Host=aws-0-us-east-1.pooler.supabase.com;Port=6543;Database=postgres;Username=postgres.db_alojamientos;Password=TU_CONTRASEÑA_AQUI",
  "ConexionReservas": "Host=aws-0-us-east-1.pooler.supabase.com;Port=6543;Database=postgres;Username=postgres.db_reservas;Password=TU_CONTRASEÑA_AQUI",
  "ConexionFacturacion": "Host=aws-0-us-east-1.pooler.supabase.com;Port=6543;Database=postgres;Username=postgres.db_facturacion;Password=TU_CONTRASEÑA_AQUI"
}
Importante sobre Supabase: Si usas Supabase, asegúrate de que el string empiece con Host=... y no con postgres://..., ya que Entity Framework de C# requiere el formato estándar ADO.NET. El puerto usual para el pooler de Supabase es 6543.

Modo Producción (Render / Azure)

NUNCA subas tus contraseñas reales a GitHub. En producción, debes crear las siguientes Variables de Entorno (Environment Variables) en el panel de control de Render:

Key (Nombre de la Variable) Value (Ejemplo)
ConnectionStrings__ConexionUsuarios
(Nota los dos guiones bajos)		 Host=...;Port=6543;Database=postgres;...
ConnectionStrings__ConexionAlojamientos Host=...;Port=6543;Database=postgres;...
ConnectionStrings__ConexionReservas Host=...;Port=6543;Database=postgres;...
ConnectionStrings__ConexionFacturacion Host=...;Port=6543;Database=postgres;...

Al usar ConnectionStrings__ (doble guion bajo), .NET en Linux/Render sobreescribirá automáticamente los valores que estén en el archivo appsettings.json.

4. ¿Cómo ejecutar el proyecto localmente?

Abre una terminal (PowerShell o CMD) en la raíz del proyecto y ejecuta:

cd Microservicio.Clientes.Api
dotnet run

La API iniciará en http://localhost:5028. Puedes abrir http://localhost:5028/swagger para probar los endpoints gráficamente.
